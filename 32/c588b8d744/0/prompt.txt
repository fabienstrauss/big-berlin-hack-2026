/agent For now, I would build only:

  1. Extraction layer with caching
  2. Unified context JSON per workspace

  No final image/video prompt generation yet.

  That split is the right boundary because it gives you a stable data pipeline
  first. Once that is reliable, the later prompt planner is just another consumer
  of the context pack.

  Step 1: Extraction Layer With Caching

  The goal is: every relevant canvas object can produce a structured “context
  artifact”, and unchanged objects do not get reprocessed.

  1. Define the extraction contract

  Create one normalized output type for all sources.

  Suggested file: app/lib/context/types.ts

  export type ContextSourceType =
    | 'note'
    | 'research'
    | 'template'
    | 'image'
    | 'video'
    | 'document'
    | 'choice-config';

  export type ExtractedContextArtifact = {
    workspaceId: string;
    sourceNodeId: string;
    sourceType: ContextSourceType;
    sourceFingerprint: string;
    title?: string;
    summary: string;
    keyFacts: string[];
    styleSignals: string[];
    audienceHints: string[];
    constraints: string[];
    claims: string[];
    ctas: string[];
    productEntities: string[];
    visibleText?: string[];
    sceneBreakdown?: Array<{
      label: string;
      timestampStart?: number;
      timestampEnd?: number;
      description: string;
    }>;
    rawMetadata?: Record<string, unknown>;
    createdAt: string;
    updatedAt: string;
  };

  Important: keep this schema broad enough for all media, but structured enough to
  merge later.

  2. Add a fingerprinting strategy

  You need a stable way to know whether a node/file changed.

  Suggested rules:

  - For text nodes: hash title + body + prompt + templateId + updated text fields
  - For uploaded files: hash storagePath + size + mimeType + lastModified if
    available
  - For choice page config: hash the full saved config JSON
  - For research/template nodes: hash the fields that affect context

  Suggested helper:
  app/lib/context/fingerprint.ts

  Use Web Crypto or Node crypto.createHash('sha256').

  3. Add Supabase table for extracted artifacts

  Create a new table, for example workspace_context_artifacts.

  Suggested columns:

  - id uuid primary key
  - workspace_id text not null
  - source_node_id text not null
  - source_type text not null
  - source_fingerprint text not null
  - artifact jsonb not null
  - created_at timestamptz not null default now()
  - updated_at timestamptz not null default now()

  Important index:

  - unique on (workspace_id, source_node_id, source_fingerprint)

  That lets you reuse the cached extraction if the source did not change.

  You may also want:

  - index on (workspace_id, source_node_id)

  Suggested file to update:
  supabase/schema.sql

  4. Build source-specific extractors

  Create one orchestrator and several extractor modules.

  Suggested structure:

  - app/lib/context/extractors/index.ts
  - app/lib/context/extractors/text.ts
  - app/lib/context/extractors/video.ts
  - app/lib/context/extractors/choice-config.ts

  Behavior:

  - text.ts
      - local only
      - no Gemini call
      - extracts fields from notes, research nodes, template nodes
  - choice-config.ts
      - local only
      - turns page-2 selections into structured artifact
  - image.ts
      - Gemini multimodal call
      - output: summary, style signals, visible text, product entities, constraints
        if inferable
  - video.ts
      - Gemini Files API upload/reference
      - output: summary, scene breakdown, pacing/style signals, visible text, CTA
        moments
  - document.ts
      - Gemini Files API upload/reference
      - output: key facts, claims, constraints, brand guidance, product entities

  5. Build a Gemini extraction client

  Do not mix this with generation code. Keep it separate.

  Suggested file:
  app/lib/providers/gemini-context.ts

  Responsibilities:

  - accept image | video | pdf input
  - prepare Gemini request
  - enforce structured JSON output
  - normalize Gemini response into ExtractedContextArtifact

  Important:

  - use server-side only
  - reuse your existing Gemini env handling
  - add robust timeout/error handling
  - return explicit extraction failure errors, not partial garbage

  6. Build a cache-aware extraction service

  Suggested file:
  app/lib/context/extraction-service.ts

  Main function:

  extractWorkspaceSources(workspaceId, nodes, choiceConfig)

  Flow:

  1. enumerate relevant sources
  2. compute fingerprint for each source
  3. check Supabase for cached artifact with same (workspaceId, sourceNodeId,
     fingerprint)
  4. if found, reuse
  5. if missing, run extractor
  6. persist artifact
  7. return full list of artifacts

  This service should be the only place that knows how to orchestrate extraction.

  7. Add source enumeration logic

  Not every node should be treated the same.

  Suggested file:
  app/lib/context/source-enumerator.ts

  Rules:

  - brainstorm nodes -> text extractor
  - research nodes -> text extractor
  - template nodes -> text extractor
  - asset nodes:
      - image asset -> image extractor
      - video asset -> video extractor
      - document asset -> document extractor
  - generation nodes:
      - probably ignore for now, unless you want prior generated outputs as
        inspiration
  - page-2 config -> choice-config extractor

  I would exclude generated outputs initially unless you explicitly want iterative
  generation from prior outputs.

  8. Add API route to trigger extraction

  Suggested route:
  app/api/context/extract/route.ts

  Input:

  {
    workspaceId: "main"
  }

  Server-side route should:

  - load canvas state from Supabase
  - load choice config from Supabase
  - call extraction service
  - return:

  {
    artifacts: [...],
    reused: number,
    generated: number,
    failed: number
  }

  Do not send raw files from browser to this route. Since your assets are already
  in Supabase, the route should fetch via stored URLs/paths.

  9. Decide when extraction runs

  For UX, I would support both:

  - manual “Refresh context” action
  - automatic extraction on:
      - file upload complete
      - note/template/research update
      - choice page save

  But at first, manual trigger is simpler and safer.

  My recommendation:

  - first implementation: manual trigger only
  - second pass: background refresh for changed sources only

  10. Add extraction status tracking

  You need visibility into what happened.

  Options:

  - simplest: return status in API response only
  - better: add per-node extraction state in UI
  - best: add a small “context freshness” indicator:
      - up to date
      - needs refresh
      - processing
      - failed

  At minimum, add a workspace-level status first.

  ———

  Step 2: Unified Context JSON Per Workspace

  The goal is: once all individual artifacts exist, build one canonical “context
  pack” that any future planner can consume.

  1. Define the workspace context schema

  Suggested file:
  app/lib/context/workspace-context.ts

  export type WorkspaceContextPack = {
    workspaceId: string;
    generatedAt: string;
    sources: {
      total: number;
      notes: number;
      images: number;
      videos: number;
      documents: number;
      templates: number;
      researchItems: number;
    };
    objective?: string;
    platforms: string[];
    requestedContentTypes: string[];
    selectedTemplates: string[];
    brandContext: {
      summary: string;
      productEntities: string[];
      claims: string[];
      constraints: string[];
    };
    creativeDirection: {
      styleSignals: string[];
      audienceHints: string[];
      ctas: string[];
      hooks: string[];
    };
    sourceArtifacts: ExtractedContextArtifact[];
    mergedContext: {
      keyFacts: string[];
      styleSignals: string[];
      audienceHints: string[];
      constraints: string[];
      claims: string[];
      ctas: string[];
      productEntities: string[];
    };
  };

  2. Build a context merger

  Suggested file:
  app/lib/context/merge-context.ts

  Responsibilities:

  - merge all artifacts into one pack
  - deduplicate overlapping strings
  - preserve source order importance
  - separate “brand/product truth” from “creative inspiration”

  Source priority I recommend:

  1. documents
  2. choice config
  3. notes
  4. templates
  5. research
  6. images/videos

  That prevents a random visual reference from overriding a brand PDF.

  3. Add deterministic merge rules

  Do not use an LLM for the merge at first. Use code.

  Examples:

  - claims
      - union of all claims
      - mark document-derived claims as higher trust
  - constraints
      - always preserve all document constraints
      - add user note constraints after that
  - styleSignals
      - collect from templates + images + videos
      - frequency-rank duplicates
  - productEntities
      - dedupe normalized lowercase form, preserve original labels
  - hooks
      - gather from notes, templates, research bullets

  This keeps the pack predictable and debuggable.

  4. Add a workspace context table

  Suggested table: workspace_context_packs

  Columns:

  - workspace_id text primary key
  - context jsonb not null
  - source_fingerprint text not null
  - updated_at timestamptz not null default now()

  source_fingerprint here should represent the combined fingerprint of all
  artifacts plus choice config. If unchanged, skip rebuilding the context pack.

  5. Build a context assembly service

  Suggested file:
  app/lib/context/context-pack-service.ts

  Main function:

  buildWorkspaceContextPack(workspaceId)

  Flow:

  1. load latest extracted artifacts for workspace
  2. load choice config
  3. compute combined fingerprint
  4. if cached context pack exists with same fingerprint, reuse it
  5. otherwise merge artifacts into new pack
  6. save to Supabase
  7. return pack

  6. Add API route for context pack

  Suggested route:
  app/api/context/pack/route.ts

  This route should:

  - ensure artifacts exist, or require extraction first
  - build or reuse the current workspace context pack
  - return the full normalized context JSON

  This will later become the single input to prompt planning.

  7. Add inspectability

  You will need to debug why context looks wrong.

  So the pack should include:

  - sourceArtifacts
  - maybe a sourceTrace section showing which node contributed what
  - trust/source type metadata if useful

  Without that, you will not know whether bad output came from PDF extraction,
  image reading, note parsing, or merge logic.

  8. Add simple UI hooks

  Not full UX yet, just enough to validate the system:

  - Extract context button
  - View context pack debug panel or modal
  - optional per-source extraction count and timestamps

  That will let you inspect the pack before you ever generate prompts.

  ———

  Implementation order

  I would do it in exactly this order:

  1. context types
  2. artifact table + context pack table
  3. fingerprint helpers
  4. source enumerator
  5. local extractors for notes/templates/research/choice config
  6. Gemini extractors for image/video/pdf
  7. extraction service
  8. /api/context/extract
  9. deterministic merge logic
  10. context pack service
  11. /api/context/pack
  12. small debug UI to inspect results

  ———

  Practical scope cut for v1

  To keep this shippable, I would narrow v1 to:

  - notes
  - templates
  - choice config
  - images
  - PDFs

  Then add video extraction right after.

  Reason: video is the most expensive and slowest modality, and it adds polling/
  file-processing complexity. The architecture should support it from day one, but
  you do not need to finish it before validating the pipeline.

  ———

  What I would avoid

  - no direct browser-side Gemini calls
  - no “one huge prompt with all raw data”
  - no free-text artifact outputs
  - no reprocessing unchanged files
  - no mixing merge logic and UI code

  ———

  Result after steps 1 and 2
  You will have:

  - cached structured context artifacts per source
  - one stable workspace context pack in Supabase
  - a clean server-side foundation for later prompt planning with Nano Banana / Veo
